Such tables are designed with multiple functionalities, enabling users to filter and search through the data efficiently. For situations involving substantial data volumes, integrating pagination with the table can significantly enhance usability by breaking the data into smaller, more manageable segments. Additionally, it's important to consider the potential delay in loading vast quantities of data. In these instances, providing clear guidance on the use and timing of loading indicators can help mitigate any user uncertainty during prolonged data retrieval times.</p>

It allows for sorting through column headers either in ascending or descending order, expanding rows to reveal more details progressively, and executing actions on rows either individually or in batches.</p>

Options for selection include both a single-select option, utilizing radio buttons, and a multi-select option, using checkboxes. This enables users to execute either individual or batch actions on the selected items. There are also usage scenarios where a switch may be used as a control  as well. </p>
      <b>Expand/Collapse</b>
      <p>The expandable variant helps present large amounts of data in a small space. Users can expand and collapse row panels to reveal and hide additional information. For nested tables, the expansion serves to allow multiple tables to be placed within master, or parent tables.</p>
      <h3>When to Use</h3>
      <ul>
      <li>To organize and display data.</li>
      <li>If your user must navigate to a specific piece of data to complete a task.</li>
      <li>Displaying all of a user’s resources.</li>
      </ul>
      <h3>When Not to Use</h3>
      <ul>
      <li>When a more complex display of the data or interactions are required.</li>

       <h3>Placement</h3>
       <p>Placement Guidelines for Tables Ensure that the tables are located within the primary content area of a page, providing ample room for the presentation of data without the need for truncation. It is advisable to refrain from embedding data tables within modals or smaller enclosures, as this can result in a cramped display or necessitate data truncation.</p>
       <p>Tables should be integrated into the page layout using one of the three gutter modes specified in the 2x grid guidelines. While tables may occupy horizontal space alongside other components and content, it is beneficial to allocate the greatest possible width to the data table. This approach facilitates the viewing of complex data by users.</p>
       <p>Illustrated through three distinct examples are the implementations of the data table in wide (default), narrow, and condensed gutter modes. Pay attention to the variations in type alignment and gutter dimensions.</p>
       <h3>Data Table Heading and Overview</h3>
       <p>The heading of the data table should clearly convey to the user the shared characteristics of the data within and its role within the interface. Beneath the heading, an optional overview may be provided to elaborate on the data's context or origin. Both the data table's heading and its supplementary overview should employ sentence-case capitalization.</p>
       <h3>Column Headings</h3>
       <p>Column headings ought to be concise, ideally limited to one or two words that summarize the column's contents. If a column heading exceeds this length, the text should be formatted to occupy two lines, truncating any overflow, with the entire heading accessible via a tooltip upon mouse hover. Sentence-case capitalization is also recommended for column headings.</p>
       <h3>Primary Action Button</h3>
       <p>When a table incorporates a primary action, it should adhere to the established guidelines regarding the use of primary buttons and the phrasing of action labels.</p>
       <h3>Cell Contents</h3>
       <p>The table header, columns and rows within the table are designed to align with interaction patterns observed in other STUDS components. This includes the use of buttons, checkboxes, radio buttons, overflow menus, search functionalities, avatars, switches, and expandable sections, all of which adhere to established guidelines for their respective behaviors.</p>
       <h3>Inline Actions</h3>
       <p>Inline actions refer to tasks that can be executed directly on individual rows of a table. These rows come equipped with an overflow menu that houses actions tailored to the particular row in question.</p>
       <h3>Meatball (Overflow) Menu</h3>
       <p>Meatball (Overflow) menu icons remain visible on each row, indicating to users that actions can be performed on the table rows.  Should the overflow menu offer less than three choices, opt to display these actions as icon buttons directly within the row. This approach eliminates an extra click and immediately showcases the available actions.</p>
       <img src="/img/meatball_overflow.png" />
       <p></p>
       <h3>Popovers</h3>
       <p>Popovers can be utilized to display more information within a cell. This scenario is available when there may more more information than can fit in a cell, or if there are other design considerations, i.e. a larger image is shown than the thumbnail that is in the cell.</p>
       <p>A user can select any component within the cell to have the popover appear. Images, chips, text, and other components can all use a popover.</p>
       <img src="/img/popover_cell.png" />
       <p></p>
       <h3>Column Selection</h3>
       <p>The Table Toolbar offers functionalities to add, delete, and lock columns. Clicking the Columns button reveals a menu where columns can be searched and selected. Users have the flexibility to choose multiple columns simultaneously, enabling dynamic addition or removal from the table. When they hit apply, those columns settings that they have set will remain.</p>
       <p>Additionally, columns can be secured by locking, a feature indicated by a lock icon positioned at the extreme right within the column selection area. Activating the lock changes the icon from an unlocked to a locked state.</p>
       <p>The user is able to select the reset button which would dynamically set the table back to it’s default column settings.</p>
       <p>The menu is closed when the user hits apply or the columns button.</p>
       <img src="/img/table_apply.png" />
       <img src="/img/table_apply_zoom.png" />
       <p></p>
       <h3>Lock and Unlock a Column</h3>
       <p>When a user wants to lock one or multiple columns, they are able to activate the lock icon on the right of the column container. The user can lock as many of the columns as they choose. This keeps the columns locked across their sessions, ensuring that then the user returns it is the same view. The icon changes state for unlocked/locked respectively.  There is the option to only lock/unlock the first 4 columns. If the first column is unlocked, then all the locked columns will be unlocked. When a user unchecks a locked column, it will unlock.</p>

The primary form of exporting utilizes the button, while the secondary funtion provides a dropdown for the user to select other formats they would like to download. The user is also able to select specific rows to only include in the export. They will accomplish this by selecting the rows they wish to export and selecting there format. The user can also export when they have filters applied. Once they have applied their desired filters, they then can go through the process of exporting to their preferred format.</p>
